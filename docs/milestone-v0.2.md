# Milestone v0.2: Read-Only Investigator (Merged)

This is the executable v0.2 specification, produced in three passes: the Codex v0.2 plan, a code-grounded joint review against `docs/roadmap.md`, and a merge of Codex's updated plan (2026-07-12), which adopted the v0.2a/v0.2b decomposition and contributed the context-calibration, retry-protocol, migration-honesty, and acceptance-gate corrections incorporated below.

The scope is unchanged: the smallest complete agentic milestone — safe workspace visibility, an iterative observe-and-decide loop, and evidence-backed engineering answers. v0.1 verification behavior and approval-gated command execution are preserved. Source disclosure to the model is treated as a **new trust boundary** with its own consent mechanism, not a capability covered by the existing security model; source modification, remote source egress, and reduced-approval autonomy remain later milestones with their own boundaries.

Delivery is two serialized, separately passing phases:

- **v0.2a — Durable substrate**: contracts, migrations, trust scoping, queue, events, cancellation, registry consolidation. Pure infrastructure; no model-facing behavior changes.
- **v0.2b — Investigation capability**: the read tier, the investigation loop, budgets, context calibration, citation validation, and the benchmark gate.

Sizing (4-6 engineer-weeks under this repository's review gates) is planning guidance, not an acceptance criterion. Each phase lands as small coherent commits; "four commits" was a review-cadence description, never an effort estimate.

## v0.2a: Durable Substrate

### Core Contracts

- Add `RunKind.VERIFICATION` and `RunKind.INVESTIGATION`. Existing API and CLI calls default to verification for compatibility. In v0.2a the investigation kind is accepted by the contracts but refused at start with an explicit "not available until v0.2b" error; the workbench begins defaulting to investigation (when a model is configured) only in v0.2b, when the loop exists. v0.2a's "no model-facing behavior changes" claim means exactly this: its MCP and UI changes are operator-facing surface, and no model-visible behavior changes until v0.2b.
- Add typed `ToolSpec`, `ToolArguments`, `ToolObservation`, `AgentDecision`, `AgentAnswer`, `SourceCitation`, `AgentBudget`, and model-usage records.
- Add terminal statuses `INCOMPLETE` and `CANCELLED`, plus operational `QUEUED` and `CANCEL_REQUESTED`, extending the existing compare-and-swap status transitions. `INCOMPLETE` (unsupported citation, budget exhaustion, any strict-decode refusal, or the repeated-no-progress stop — the enumerated causes defined in v0.2b) is distinct from `FAILED` so benchmark and telemetry pass rates are not polluted by budget stops.
- Replace adapter `tools()` duplication with one registered command-tool definition containing metadata, exact argv, and its `CommandRule`. Generate runner policy from these definitions.
- **Safety invariant:** rules derive from static code constants only. Workspace detection may *select* an available static rule; it may never create one, parameterize one, or weaken one — workspace content must never influence `requires_approval` or rule identity.
- Use the consolidation to fix verified drift: remove the dead `pytest`/`ruff-check` rules in `policies.py`, remove unimplemented `tools()` metadata entries, and reconcile the Django adapter's `tools()`/`profile()` name mismatches. Add an invariant test proving every advertised tool can dispatch and every command rule belongs to exactly one tool.

### Trust and Persistence

- Introduce scoped workspace attestations: `source_read` and `code_execution`. Existing trust records migrate to `code_execution` only — the conservative reading of the current UI consent text, which speaks solely of executing code — so no prior attestation silently gains source-disclosure authority.
- **Migration mechanics (downgrade-safe physical schema):** scoped attestations live in a **new `workspace_attestations` table**, and the legacy `trusted_workspaces` table is dropped (or left empty) by the migration. This is deliberate: a v0.1 binary has no schema-version check, selects by workspace alone, and treats any surviving row as execution trust (`service.py:364-375`, `:487-490`) — rebuilding the old table as `(workspace, scope)` would let a `source_read`-only row authorize code execution under v0.1 after `code_execution` is revoked. With the legacy table empty, v0.1 fails closed. A regression test executes v0.1's exact trust query against a migrated state directory containing only `source_read` and asserts it authorizes nothing.
- **Migration execution:** a migration coordinator runs after taking the state-directory lease and before any store opens. Migrations are transactional per database, **forward-only**, and guarded by `PRAGMA user_version` in every database file; because state spans multiple SQLite files (runs, trust, replay, checkpoints), the coordinator maintains a version vector, takes a WAL-consistent pre-migration backup via the SQLite backup API (not a file copy), and writes a completion marker only after every database commits. **Crash recovery is defined:** at startup, backup-present-without-marker means an interrupted migration — the coordinator restores every database from the verified backup and re-runs the migration; marker-present means the backup is disposable; mixed versions *without* a backup (the only unrecoverable shape) fail closed with a named manual-restore instruction. Mixed-version refusal therefore never strands state that the coordinator itself produced. Replace all ad hoc `PRAGMA table_info` column patching with these versioned migrations covering runs, trust scopes, events, work items, budgets, final answers, usage, and cancellation. New binaries fail closed with an explicit version error on an unrecognized future schema; downgrades are documented as unsupported. Existing verification runs and checkpoints are preserved.
- The scope enum is designed to admit a future `model_egress` scope (v0.3) so per-workspace cloud consent lands on this storage rather than a new mechanism.
- Permit `source_read` only with a loopback model endpoint in v0.2. Remote source-bearing investigation is rejected; the per-workspace `model_egress` attestation scope that would permit it is deferred to v0.3.
- **Loopback is verified at the socket, not the string:** current endpoint validation accepts the literal hostname `localhost` lexically and never checks the connected peer (`planner.py:69-81`, `:197-212`). For source-bearing requests, either restrict configuration to numeric loopback literals or verify `getpeername()` is a loopback address after connecting and before sending bytes. The model/endpoint fingerprint and loopback eligibility are persisted with the run and revalidated at every dequeue and resume — a durable run must not silently continue against a different endpoint than the one consented to.
- **Scope enforcement points:** advisory investigation requires `source_read`; assisted investigation requires **both** `source_read` and `code_execution`. Scopes are rechecked at worker dequeue and before every source-bearing model request, workspace read, and command dispatch — not only at run start. Revocation invalidates queued work items for that workspace and sets `CANCEL_REQUESTED` on its active runs; its linearization point is the scope-generation check at the next enforcement point, and the revocation tests assert exactly that.
- **Scope coverage for commit review:** a *direct* `review-commit` CLI invocation by the operator remains explicit per-run source-disclosure consent and keeps working without new attestations. *Agent-triggered or automatic* review requires the workspace's `source_read` attestation — the distinction is who decided to disclose source: a human at the keyboard, or the loop.
- Add scoped trust revocation through HTTP, CLI, and the workbench (no revocation path exists today). The consent UI distinguishes "disclose source to a local model process" from containment: per `docs/safety.md`, a loopback model server is still an untrusted peer.

### Background Runs, Events, and Cancellation

- Add a service-owned single-worker queue so start and approval endpoints return `202` without blocking SQLite/LangGraph execution.
- **Resumes route through the same worker:** `approve_and_resume` currently re-enters the workflow synchronously on the HTTP thread (`control_plane.py:224-238`). The checkpointer serializes individual checkpoint reads/writes with an internal lock, but a queue that owns only starts would still leave two concurrent `graph.invoke` executions interleaving at node/checkpoint granularity and contending with the runs-table CAS transitions. The queue serializes whole run executions so LangGraph's shared SQLite connection has one writer. Starts and resumes serialize through the worker; decline never touches the graph, so it is an immediate terminal CAS on the HTTP thread and does not enter the worker.
- **Approval guarantees stay synchronous:** the CAS status claim, pending-approval digest match, and single-use consumption stance execute transactionally on the HTTP thread at enqueue time — they are cheap SQLite writes; only graph execution moves to the worker. A stale digest or duplicate approval therefore still fails synchronously with a conflict response; `202` means "validly claimed and queued".
- **Challenge identity (anti-replay across iterations):** because the loop legitimately repeats identical commands, the action digest alone (workspace/domain/rule/argv, `approvals.py:75-89`) no longer uniquely identifies a decision. Every approval challenge carries a durable `challenge_id` bound to the run ID and action ordinal; an approval must present and atomically consume that `challenge_id` together with the digest, and the `challenge_id` is included in the signed claim. A delayed or duplicated approval POST for action N can therefore never authorize an identical action N+1. Approval POSTs are never automatically retried by any client.
- **Consent freshness:** the human's decision timestamp is persisted as `approved_at` with an immutable `grant_expires_at` (the existing 300-second TTL measured from the human's decision, not from dequeue). The signed token is minted at worker dequeue for execution purposes but must not outlive `grant_expires_at`; an approval whose grant expires while queued returns the run to a fresh challenge rather than executing on stale consent. The worker re-verifies run status and attestation scopes immediately before invoking the graph.
- **Durable work items:** `QUEUED` status alone cannot reconstruct what was queued. A transactional `run_work_items` table records operation kind (start / approved-resume / decline), run ID and action ordinal, actor identity, the exact challenge consumed, timestamps, grant expiry, and claim state — so a crash between the CAS and the in-memory enqueue is recovered from the table, not lost. Decline is an immediate terminal CAS on the HTTP thread and does not need the worker.
- **Restart reconciliation:** `QUEUED` runs found at startup re-enter the queue; they are never marked failed for lacking a checkpoint. Genuine approval interrupts are preserved exactly as today. A run that was executing a command when the process died is conservatively marked an unknown-outcome failure, as today.
- Persist ordered run events for status changes, model calls, tool lifecycle, approval requests, usage, completion, and cancellation. Add authenticated bounded long polling: `GET /runs/{id}/events?after=<sequence>&wait_seconds=<n>`, returning ordered JSON event batches and a next cursor. Lifecycle events only; command output appears after bounded redaction completes, and raw stdout is never streamed. This replaces any SSE ambition for this milestone.
- Add `POST /runs/{id}/cancel`. Queued and approval-waiting runs cancel immediately. **Cancellation semantics stated honestly:** a dispatched subprocess blocks inside the graph node until it exits or times out; cancellation of a running command is node-boundary-only in v0.2. `CANCEL_REQUESTED` is checked at every node boundary, and the existing per-command timeout remains the bound on a hung subprocess. Immediate process-tree termination of in-flight commands is deferred alongside streaming output.
- **Cancel/approve race semantics:** cancel of a `QUEUED` or approval-waiting run is a CAS transition on the HTTP thread that also clears the pending approval challenge, invalidating its digest. The worker re-CASes run status immediately before graph invocation and drops enqueued starts/resumes whose run is `CANCELLED` or `CANCEL_REQUESTED`. A concurrent approval that loses the CAS race receives the standard stale-challenge rejection. These are the outcomes the cancellation-race tests assert.
- **202 contract breaks owned:** the CLI `start` and `approve` commands' exit codes (including exit `2` for waiting-on-approval) derive from a synchronous final status today, and a direct CLI invocation owns an ephemeral service with no surviving worker (`cli.py:55-87`) — so `--no-wait` is only meaningful when submitting to a running control plane, and is refused (with an explanatory error) for direct local invocations, which remain synchronous by construction. Both `start` and `approve` keep their existing exit-code contracts by default. Update the MCP `start_run` docstring, which becomes false the moment start enqueues.
- Update the workbench UI: live activity from the event feed, budget counters, evidence citations, final answers, retry-safe reconnect, source-consent state, and a cancel control.
- Add MCP `list_runs`, `get_plan`, and bounded `get_trace` through an **MCP-specific projection — the HTTP-safe projection is not MCP-safe**: the HTTP view legitimately shows the operator absolute workspace paths and pending-approval challenges, both of which must stay out of MCP. Every existing MCP method serializes the full `RunRecord` (`mcp_server.py:35-62`) — the `trace_path` leak affects `start_run` and `create_run`, not just `get_run` — so the MCP projection is applied to create/start/get/list alike and excludes approval challenges, absolute paths, source observations, and source-bearing output. MCP `create_run` defaults `RunKind` to verification. The MCP projection applies to **every** MCP tool, not only run methods: `profile_workspace` currently returns the complete `WorkspaceProfile` including the absolute root, absolute command executables, and toolchain paths (`mcp_server.py:27`) — the MCP variant returns workspace-relative or opaque identifiers only. Trust, approval, and source content stay out of MCP.
- **Restart/cancel/retry state machine:** the terminal mappings are fixed now — terminal protocol failure → `FAILED` with stop reason `model_protocol`; the repeated-no-progress stop → `INCOMPLETE` with stop reason `no_progress` (a fourth enumerated `INCOMPLETE` cause alongside unsupported citation, budget exhaustion, and strict-decode refusal); `CANCEL_REQUESTED` found at startup → `CANCELLED` (the cancel wins; queued work items for that run are dropped); crashes inside model-call, read, compaction, and final-answer nodes are safely retried on recovery because none have side effects, while commands keep today's unknown-outcome conservatism. The complete CAS matrix — every status × work-item kind × in-flight node, both cancel/approve linearizations, idempotent cancel — is a **required, review-blocking deliverable of the first v0.2a commit**, and the race tests are written against it.
- Add CLI parity for list, trace, decline, cancel, scoped trust, and revocation.

## v0.2b: Investigation Capability

### Safe Workspace Tools

Register three parameterized, pure-Python tools: `workspace.list_files`, `workspace.search_text`, and `workspace.read_file`.

These are the project's first **model-facing** source tools — not its first workspace content reads: adapter profiling already reads workspace file content before any consent (the PyTorch adapter byte-sniffs up to 1 MiB, `adapters/pytorch.py:49-55`, reachable from CLI, HTTP, MCP, and workflow profiling, and can itself traverse reparse-backed content). v0.2b therefore also makes pre-consent workspace *detection* metadata-only (names, sizes, markers), and any profiler that must read content routes through this same attested reader. The three tools qualify for the unattended tier under the roadmap's standing invariant: pure Python, no workspace-controlled configuration that alters execution, gated by the workspace's `source_read` attestation with a loopback-only model endpoint. That classification rule — not convenience — is what admits a tool to the unattended tier.

- Accept only workspace-relative paths. Reject absolute paths, traversal, VCS internals, environment/credential/key files, binary content, hard links, DOS device names (`CON`, `COM1`, `NUL`, …), NTFS alternate-data-stream syntax, trailing-dot and trailing-space filename aliases, and **every symlink, junction, or reparse point in v0.2** — no benign-tag whitelist yet; OneDrive placeholder-backed workspaces are therefore refused rather than risking silent hydration (a hidden network side effect on a loopback-only milestone), and a tag whitelist is a documented later refinement.
- **Per-domain deny patterns (from the roadmap):** beyond generic env/credential/key files, deny-or-redact `*.keystore`, `*.jks`, `google-services.json`, provisioning profiles, `local.properties` credential lines, and private-key material of any extension. Each future domain pack contributes patterns to this single policy.
- **Windows mechanics:** win32 has no `O_NOFOLLOW`, and the repository's existing check-then-open idiom is itself a TOCTOU. Traversal is **handle-relative and race-free**: the workspace root is opened once, and every subsequent component is opened *relative to its retained parent directory handle* with `FILE_FLAG_OPEN_REPARSE_POINT` — never by re-walking an absolute path — so a parent directory swapped mid-traversal cannot redirect the walk, and a reparse point on any component is opened (not resolved) and refused with nothing followed and nothing hydrated. Equivalently, an NT-level open with `OBJ_DONT_REPARSE` (whole-path reparse refusal) is an acceptable implementation. After the target opens, validate the final handle's fully resolved path against the workspace root and compare handle file identity with the pre-open `lstat`. Detect concurrent mutation by comparing file identity, size, and mtime on the same open handle before and after the read (a single mid-read hash cannot detect it); any mismatch refuses the observation. Enforce a regular-file-only, link-count-of-one rule. Enforce deny patterns case-insensitively against canonical names, including 8.3 short-name aliases. Never deliver lossily decoded bytes to the model — strict-decode UTF-8 and, on failure, refuse the read (this is deliberately *stricter* than commit review, which replacement-decodes blob text into the diff while forcing `INCOMPLETE`).
- **Executable tool contracts (exact):** `workspace.read_file{path: str ≤512 chars, start_line?: int ≥1, max_lines?: int ≤200}`; `workspace.list_files{path: str ≤512, glob?: str ≤128 (literal segments + `*`/`**` only), cursor?: str}`; `workspace.search_text{query: str ≤256 (bounded literal — no user-supplied regex in v0.2), glob?: str ≤128, cursor?: str}`. Every tool call has a 10-second deadline. Listings and search results have per-entry caps (path ≤512 chars, snippet ≤200 chars) and post-serialization ceilings (~2,000 estimated tokens per response, byte backstop 16 KiB), with continuation via cursor — 500 long paths or one 1 MiB line must not be able to exceed the observation budget.
- Return bounded relative paths, numbered lines, content SHA-256, truncation state, redaction state, and an observation ID. **Persist every model-visible observation — complete or incomplete — in the durable event log**, exactly as sanitized (post-bounding, strict decode, redaction, neutralization); raw bytes are never persisted, and the raw-content hash plus invalid-span masks are stored alongside. Redaction and neutralization must be **line-preserving** so persisted line numbers stay valid (the current private-key redaction collapses lines, `redaction.py:45-55` — the read tier's replacement preserves newline count or stores an explicit line map). Final citations must resolve to an earlier observation's path, content hash, and returned line range, including the uncited-but-valid spans of incomplete observations, and must survive reconnect and compaction.
- Apply secret redaction and line-preserving workspace-instruction neutralization before content reaches the model.
- **Observation incompleteness vs run verdict (this rule appears identically in the Investigation Loop section):** bounded truncation with elision markers is a *normal* observation property — every large file truncates by design — visible to the model and reflected in the final answer's completeness field, and it does not by itself affect the run verdict; the loop's remedy is a narrower re-read. Secret redaction and instruction neutralization mark the observation incomplete and invalidate any citation anchored to a redacted or neutralized span. Run-level `INCOMPLETE` is forced only by enumerated, **mechanically decidable** conditions: an unsupported citation in the final answer, budget exhaustion, any strict-decode refusal during the run ("a read the answer's evidence requires" cannot be inferred from model prose, so every strict-decode refusal counts; typed binary-detection refusals do not — the model is told the file is binary and simply cannot cite it), or the repeated-no-progress stop. This deliberately diverges from commit review's all-or-nothing `INCOMPLETE` verdict: a one-shot verdict cannot compensate for missing bytes, but an interactive loop can re-read around them — so the fail-closed unit here is the citation, not the whole run.
- Extract only reusable bounding/redaction primitives from commit review; do not reuse its immutable-Git reader for mutable working-tree access.

**Bounds (token-denominated, derived from measured context):**

- Each file: 1 MiB on disk maximum to be readable at all.
- Each read: 200 lines and ~3,000 estimated tokens (~12 KiB backstop), with head/tail elision markers on truncation.
- Listings: 500 entries. Searches: 100 matches over at most 2,000 files / 8 MiB scanned, snippet lines capped.
- Prompt capacity is an explicit equation, with `C` the calibrated context length: `H_max = max(0, min(C/2, C − completion_reserve − non_observation_tokens − safety_margin))`, where `completion_reserve` is the next request's escrowed allowance, `non_observation_tokens` counts everything else actually in the prompt (system prompt, goal, tool descriptions and schemas, sanitized profile, observation catalog — which grows, pinned compacted notes, any retry-correction message), and `safety_margin` absorbs the estimator's error band. Compaction triggers when estimated history exceeds **90% of `H_max`** and compacts down to **60% of `H_max`** — "approaches the limit" is not a trigger. `safety_margin` is defined as `max(0.1 × C, 2 × max_observed_estimator_error)` from calibration. The token estimator is calibrated as a conservative **upper bound** against endpoint-reported usage, not an average. The 512 KiB total-returned-source byte budget remains as a defensive backstop.
- **Command-output distillation:** in assisted mode a frozen command can return up to 2 MB of redacted stdout, which would defeat every read budget. Command observations are distilled to exit status, classified error lines, and bounded head/tail content (~3,000 estimated tokens). Full redacted output is retained only in the trace.

### Investigation Loop

- Add a separate durable LangGraph investigation workflow: `profile → decide → dispatch → observe → decide`, terminating on finish, refusal, cancellation, terminal protocol failure, or budget exhaustion. The v0.1 plan-once verification graph is untouched.
- **Retirement clause:** the side-by-side graph is accepted for exactly one milestone. When v0.3 adds patches and therefore approvals, the investigation graph absorbs the approval machinery and the plan-once graph is deleted or frozen behind a legacy flag in that same milestone. Two topologies must not drift indefinitely.
- Each model turn selects exactly one registered tool or emits a structured final answer. Repeated tools are allowed with new arguments; identical no-progress calls are stopped after two repetitions.
- Supply the sanitized workspace profile, typed tool descriptions, compact prior activity, and bounded recent observations. Never expose absolute paths, credentials, raw policy objects, or hidden reasoning.
- **Observation catalog:** the prompt always carries a deterministic compact catalog of every observation so far — IDs, relative paths, content hashes, tool names, statuses — maintained by the runtime, not the model. Model-produced summaries (including compaction output) are non-authoritative and never citable; citations resolve only against the durable event log.
- Treat command failures as observations so the model can diagnose and replan. Security-policy violations remain immediate refusals.
- Advisory investigation permits read tools only. Assisted investigation permits read tools plus the existing frozen command tools, with a fresh human approval interrupt for every execution. `BOUNDED_AUTO` and `WORKFLOW_AUTO` remain approval-gated and hidden in the UI until an OS sandbox exists.
- **Call budgets:** 12 logical decisions and at most 18 physical model requests per run. The decision budget decomposes explicitly: at most 10 decisions may select tools (that is the tool-call budget) and the remaining 2 cover the final answer plus one recovery turn after a stalled or refused action — so the arithmetic closes: 10 tool decisions + ≤2 answer/recovery decisions = 12, and "the full decision budget remains reachable" means exactly this decomposition. Each decision gets at most three attempts: the original request, at most one transport retry, and at most one schema retry. Compaction requests count against the physical budget, not the logical one. Also budgeted: 4 command calls (within the 10 tool calls), the observation-history capacity above, 24,576 total completion tokens, and 600 active seconds excluding approval wait. Overrides validate against hard server caps.
- **Completion-token composition (escrowed):** each request asks for `min(4096, floor(remaining_completion_budget / remaining_logical_decisions_including_final_answer))`, with a floor request of 1,024 — so completion capacity is escrowed across the decisions still owed and 12 decisions remain reachable even under conservative charging (a bare `min(4096, remaining)` rule would let requested-token charging exhaust 24,576 in six calls when the endpoint omits usage). Every call produces a typed call-result record: requested allowance, validated reported usage and model ID, latency, and failure class (the current client discards the usage envelope entirely, `planner.py:223-254`). GPT-OSS reasoning tokens count against completion, so the effective JSON budget per call is smaller — the bounded retry, not schema hope, handles mid-JSON truncation.
- **Strict response protocol (no repair):** each response must be exactly one parseable, schema-valid JSON object. The client does not strip prose, fences, or reasoning prefixes and does not extract the first balanced object — silent syntactic repair would hide protocol garbage and accept smuggled content. A malformed response consumes that decision's schema retry with a corrective message; a second failure is a terminal protocol failure for the run.
- **Retry accounting:** retries are charged against the physical-request, token, and active-time budgets. Raw failures, recovered retries, and terminal protocol failures are recorded separately in usage records — recovered instability must be visible, never silently absorbed.
- Conservatively charge the *requested* completion tokens — the possibly reduced per-call `max_tokens`, not the constant cap — when an endpoint omits usage. Budget exhaustion returns `INCOMPLETE` with a precise stop reason.
- Require final answers to contain a summary, findings, recommended next actions, completeness, and validated source citations. **Verdict rule (identical to the Safe Workspace Tools statement):** run-level `INCOMPLETE` is forced only by an unsupported citation, budget exhaustion, any strict-decode refusal during the run, or the repeated-no-progress stop; routine bounded truncation is a normal observation property remedied by narrower re-reads, typed binary-detection refusals are ordinary observations, and redacted or neutralized spans are simply uncitable.
- Restrict unattended source-bearing investigation to loopback endpoints throughout v0.2.

### Context Calibration and Model Reliability (gate prerequisites)

Without these items the exit gate measures endpoint stability, not investigation ability:

- **Measured context calibration (replaces any asserted context figure):** a calibration workflow tests 16K, 24K, 32K, and 48K against the actual LM Studio model and GPU configuration. Each point must show no model-server crash or OOM, at least 1.5 GiB VRAM headroom, and 20/20 strict-schema probe responses without retry. The largest passing context is selected; falling back to 16K is legitimate when it is what passes. The benchmark report records model identity, LM Studio version, VRAM usage, latency, and the selected context length. All prompt and observation budgets derive from the measured capacity, not from an assumed number.
- **History compaction:** at the capacity formula's high watermark, the local model summarizes older turns into a pinned investigation-notes block, compacting down to the low watermark. The observation catalog and durable event log survive compaction untouched; compaction summaries are non-authoritative and never citable, so compaction can never manufacture or orphan evidence. Compaction requests count against the physical budget and use the same bounded-retry rule.
- **Bounded retry (specified in the Investigation Loop):** at most one transport retry and one schema retry per decision, fully accounted. There is deliberately no JSON repair layer.
- **Token estimator:** chars-per-token heuristic calibrated against endpoint-reported usage as a conservative **upper bound** (never an average); used by every read-tier and history cap above.

### Observability

- Record model-call counts (logical and physical), reported prompt/completion tokens, endpoint-reported model ID, latency, tool counts, read bytes, active duration, stop reason, raw failures vs recovered retries vs terminal protocol failures, and optional configured monetary cost.
- Add structured standard-library logging keyed by run ID and event type, excluding goals, source text, command output, tokens, and secrets.
- Preserve complete redacted traces while keeping browser and MCP projections bounded.

## Verification and Benchmark

- Adversarial tests: traversal, junctions and reparse points, hard links, alternate data streams, 8.3 names, trailing-dot/space aliases, device names, mid-read races, sensitive files, binary/encoding failures, line-preserving redaction (line numbers survive), prompt injection, search limits and serialized-output ceilings, unsupported citations, citation-to-catalog integrity across compaction and reconnect, registry drift, loop stalls, budgets (logical vs physical accounting, escrow arithmetic), approval replay including the identical-repeated-command challenge-ID case, consent expiry of a queued approval, v0.1's exact trust query against a migrated store containing only `source_read` (must authorize nothing), scope revocation racing queued and active work, migration (forward, WAL-consistent backup, unknown-version refusal, multi-database version vector), queued-run restart requeue, `CANCEL_REQUESTED` startup disposition, writer serialization, event ordering and cursors, cancellation races, CLI `--wait` exit-code compatibility, MCP projections on create/start/get/list, and command distillation.
- Scripted-planner integration tests covering read → command failure → observe → replan → finish, cancellation during a long subprocess (asserting node-boundary semantics), and compaction preserving citation validity across a 20+ step synthetic loop.
- Package seven investigation tasks: Android, iOS, C/C++, Django/full-stack, PyTorch, generic architecture research, and approval-gated generic Git observation. The read tools are domain-independent, so cases run under the fixture repository's detected domain; the C/C++ case runs under `Domain.GENERIC` in v0.2 — `Domain.CPP` does not exist yet and lands in M3 per the roadmap. **Each case has three equivalent goal variants**, scored deterministically against required concepts, validated evidence anchors, required tool use, completeness, budget compliance, and endpoint-model consistency, with a written rubric for what constitutes an evidence-backed answer.
- **Exit gate:** with local GPT-OSS-20B at the calibrated context, every case must pass at least two of its three variants, and the aggregate must reach at least 19/21 — which guarantees at least one complete 7/7 suite while also measuring robustness to goal phrasing. **Gate-fatal regardless of the 19/21 arithmetic** (never countable among the tolerated failures): any unsupported citation, path-policy violation, unredacted secret, unrecovered model-protocol failure, endpoint-model mismatch on any successful response, or an actual budget-cap breach. Compliant budget exhaustion (a correct `INCOMPLETE` stop) is an ordinary variant failure; *exceeding* a cap is an integrity violation. A variant that passes only after permitted retries counts as a pass, with its retries reported; recovered transport instability is reported independently and cannot be counted as capability failure or silently hidden. The approval-gated Git case uses a scripted approver that issues each per-action approval through the real control-plane endpoint — reproducible without creating any standing approval. Once v0.3 cloud clients exist, run the same 21 variants once against a frontier model as an informational calibration point, not a v0.2 gate.
- Run full Pytest coverage, Ruff, formatting, strict Mypy, lock/distribution checks, cross-platform CI, and desktop/mobile browser QA with no overlap or console errors.

- **Executable review sequence (one order, no contradictions):** for each commit — Codex implements; tests run locally; an independent Codex reviewer audits the local diff; the commit is pushed to main (no force-push); the public-commit reviews then run — Inverse-Agent's own `review-commit` and Claude CLI with exact `--model claude-fable-5 --effort max` (the installed CLI's maximum effort level — prior review-log entries recorded the request as "ultracode", which is not a valid `--effort` value on Claude Code 2.1.206; verify `modelUsage` contains `claude-fable-5`, disclose additional models). Blocking findings from the post-push reviews are fixed forward in follow-up commits, and a phase is not declared complete until its final commit's public reviews PASS. This matches the repository's demonstrated practice (private-worktree review was previously refused; `db231f7` was reviewed post-push and fixed by `5d9c17e`). Reviewing a *local* diff with Claude instead requires explicit disclosure consent per event, and is not the default.
- Record outcomes in `docs/review-log.md`. No release, tag, or published package.
