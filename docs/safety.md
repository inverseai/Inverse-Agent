# Safety Policy

## Threat Model

The planner, model endpoint, and workspace content are untrusted. The approval signer, runner policy, authenticated control plane, and local state directory are trusted. Model output may select only registered tools; it cannot supply an executable, argv, approval bit, or approval token.

## Enforced Controls

- A workspace must be explicitly attested as trusted before assisted or automated execution. Advisory planning does not execute code.
- Command rules match exact normalized argv. Unknown flags, additional arguments, alternate scripts, and output paths are refused.
- The runner executes its own resolved absolute executable, never the caller's executable string.
- Workspace-local executables are allowed only for approval-gated rules.
- Each capability is signed and bound to the resolved workspace, domain, rule, and argv. Capabilities expire and are consumed once through a SQLite uniqueness constraint.
- The control plane fails to start without distinct operator and approver credentials. Human identity comes from server configuration, not request text. Only `/health` and the exact same-origin UI assets are public, and the CLI binds the server to loopback.
- The child environment is allowlisted and does not inherit Inverse-Agent credentials.
- Output capture is bounded, decoded defensively, and redacted before it enters traces.
- Timeout cleanup targets process groups on POSIX and process trees on Windows.
- Model endpoints are loopback-only by default. Remote inference requires dual operator opt-in and HTTPS. Redirects and environment proxies are disabled, responses are size-capped, and endpoint failures never trigger a deterministic fallback.
- Model API keys are environment-only configuration and remain excluded from workspace subprocess environments, traces, fingerprints, and startup summaries.
- The browser receives only safe run projections and bounded, re-redacted trace previews. Absolute state paths, capability IDs, and command internals outside the human approval view remain server-side.
- The same-origin workbench uses strict browser security headers, no CORS or cookies, separate operator and approver authorization, and no dynamic HTML execution sinks.
- A process-lifetime OS file lease enforces one service writer per state directory, so restart recovery cannot race an active planning or approval operation.

## Tool Hardening

Git commands use exact read shapes with global/system configuration, optional locks, terminal prompts, pagers, and fsmonitor disabled. Generic Git actions still require approval because `git status` can invoke clean/filter helpers configured by the repository. Gradle commands use the absolute project wrapper, run offline, and always require approval because configuration evaluates project code. Django, PyTorch, CMake, and Xcode actions likewise require approval.

Commit review uses a narrower Git surface than generic status. It accepts only hexadecimal commit IDs, discovers Git only beneath operating-system installation roots (machine-registry Program Files roots on Windows), rejects linked or junction-backed executable paths, disables replacement objects, lazy object fetching, and commit graphs, enumerates raw tree changes, reads size-checked blobs through bounded pipe readers, and creates diffs internally. Raw old/new modes make mode-only changes visible. Byte changes that collapse under logical-line rendering are summarized with line-ending metadata and force `INCOMPLETE`; global kind comparison, aligned logical-line comparison, and mixed-kind insertion/deletion checks cover new mixed-EOL files, EOL conversion, and same-kind redistribution. LF, CRLF, and CR-only line counts share the same byte-record parser. Any non-UTF-8 source or metadata decode loss also fails closed instead of allowing a complete verdict over replacement text. Overflow or timeout kills Git, force-closes both pipes, and joins both readers before returning. It requires an in-workspace `.git` object store and rejects links, common directories, alternates, grafts, and shallow ancestry. It never invokes repository hooks, clean/smudge filters, text conversion, or external diff commands. Gitlinks are identified from raw mode `160000` and represented with their tree-entry object IDs without dereferencing submodule commits. Candidate model findings use opaque file IDs and must supply verbatim evidence plus its added/removed side; Inverse-Agent maps the longest unique changed line mechanically. Exact source matching precedes an optional one-character declared-side marker fallback, so diff-prefixed model quotations do not invalidate removed-line findings and real increment/decrement source remains distinct. Deterministic findings may inspect a raw blob locally, but their model-facing evidence is re-derived from the redacted diff. Deterministic candidates consume the bounded candidate budget before scout hypotheses, but they are presented only when model adjudication accepts them; any unique candidate displaced by the bound forces `INCOMPLETE`. Detectors provide explicit root-line sets; accepted model restatements collapse only when a shared defect category anchors to one of those roots or copies the same evidence. Android bridge proof is receiver-specific; C++, iOS, and Django checks are lexically scoped; and PyTorch source contradiction uses ordered, overwrite-aware provenance. Every model candidate field is redacted again before adjudication. Any secret redaction from source diff or imported context forces `INCOMPLETE`, preventing omitted source from producing `PASS`. Pure-comment review instructions are neutralized; ambiguous instruction-like executable source is omitted and forces `INCOMPLETE`. The report records source, metadata, context, goal sanitization, and candidate-budget truncation. Python context includes namespace submodules, bounded containing and ancestor package initializers, package-attribute initializers, imported modules, and unchanged imported symbols from files that also changed; a partial symbol extraction is retained but forces `INCOMPLETE`. Omitted changed bytes, unresolved imported symbols, and bounded context/dependency overflow all force `INCOMPLETE`.

## Residual Risks

Inverse-Agent does not yet provide an OS-level network namespace, filesystem virtualization, or container boundary. A trusted workspace can still perform any action available to its process identity after a human approves execution. Strong isolation requires a VM, container sandbox, dedicated build host, or CI runner.

Approval binds the action, not every transitive file imported by a build system. Operators should avoid modifying a workspace while an approval is pending. The service refuses state directories under the workspace root; production deployments should additionally protect state with OS ACLs.

Redaction is defense in depth, not a proof that arbitrary secrets can be detected. Raw logs and source remain local by default; external inference or artifact upload should use explicit egress policies.

Commit review intentionally sends redacted source diffs to the configured model endpoint. Loopback inference keeps that data on the workstation. Enabling a remote endpoint is an explicit code-egress decision and still requires the existing environment-plus-CLI dual opt-in. Source-instruction neutralization covers common review-directed patterns but is not a proof against every prompt-injection form; opaque filenames, local JSON Schema validation, changed-line enforcement, and non-executable findings contain the remaining impact.

A loopback model server is still an untrusted network peer. A compromised local process can impersonate it, return malicious plans, or observe planning prompts. Response size, JSON decoding, recursion, oversized-number, and local JSON Schema failures are contained as model protocol errors. Exact tool validation and human approval remain mandatory at every autonomy level, including bounded and workflow automation.

The model transport reapplies a shrinking monotonic deadline before response headers and every body chunk. Python's standard HTTP header parser may perform multiple socket reads under the remaining header timeout, so a deliberately slow opted-in remote endpoint can still extend one planning call. Keep inference loopback-local where possible and supervise long-running service processes externally.
